# ISSUE 1 — Temporary Request Error

## 1. Issue Title
**Missing `temporary_access_days` Column Causes SQL Fatal Error on Temporary Access Support Ticket Submission**

## 2. Error Category
**DB Schema / Backend ORM**

## 3. Technical Root Cause Analysis

The controller `NewSupportForAsttrolokController::store()` at line 497 explicitly sets:

```php
if ($request->support_scenario === 'temporary_access') {
    $data['temporary_access_days'] = 7;
    $data['temporary_access_reason'] = $request->temporary_access_reason;
}
```

And the validation rules at line 353 require:

```php
case 'temporary_access':
    $rules['webinar_id'] = 'required';
    $rules['temporary_access_days'] = 'required';
    $rules['temporary_access_reason'] = 'required';
    break;
```

However, the original migration (`2026_01_16_114513_create_new_support_for_asttrolok_table.php`) **never defined** the `temporary_access_days` or `temporary_access_reason` columns. The migration creates columns for extension, mentor, relative, and other scenarios but omits the temporary access scenario fields entirely.

When Eloquent executes `NewSupportForAsttrolok::create($data)`, the generated SQL includes `temporary_access_days` in the INSERT column list. MySQL rejects this with:

```
SQLSTATE[42S22]: Column not found: 1054 Unknown column 'temporary_access_days' in 'field list'
```

## 4. Why This Error Occurs

**Root cause: Migration gap.** The controller and model (`$fillable` array includes `temporary_access_days` and `temporary_access_reason`) were developed expecting these columns, but the migration file was never updated to include them. This is a classic schema-code desync caused by:

1. Developer added the feature logic in controller + model
2. Developer forgot to add the columns to the migration
3. No CI/CD schema validation exists to catch the mismatch
4. No integration test covers the temporary access ticket creation path

## 5. Which Layer Is Failing

| Layer | Status |
|-------|--------|
| **Frontend (Blade form)** | ✅ Working — form submits `temporary_access_days` field |
| **Middleware** | ✅ Working — request reaches controller |
| **Controller validation** | ✅ Working — validation passes |
| **Controller logic** | ✅ Working — builds correct `$data` array |
| **Eloquent ORM** | ✅ Working — generates correct INSERT SQL |
| **MySQL Schema** | ❌ **FAILING** — column does not exist |

**Failing layer:** Database Schema

## 6. SQL Fix

```sql
ALTER TABLE new_support_for_asttrolok
    ADD COLUMN temporary_access_days INT NULL AFTER extension_reason;

ALTER TABLE new_support_for_asttrolok
    ADD COLUMN temporary_access_reason TEXT NULL AFTER temporary_access_days;
```

**Applied:** ✅ Both columns added and verified.

## 7. Backend Fix Example

No backend code change needed — the controller and model were already correct. The only fix is the SQL schema change above.

However, for defensive coding, the controller should wrap the create call:

```php
try {
    $supportRequest = NewSupportForAsttrolok::create($data);
} catch (\Illuminate\Database\QueryException $e) {
    Log::error('Support ticket creation failed', [
        'scenario' => $request->support_scenario,
        'error' => $e->getMessage(),
    ]);
    return back()->withErrors(['error' => 'Unable to create support ticket. Please contact support.']);
}
```

## 8. Security Risk

- **Medium.** With `APP_DEBUG=true`, the raw SQL error exposes:
  - Full table name and structure (`new_support_for_asttrolok`)
  - Database column names
  - Full SQL query with user-submitted data
  - Internal file paths via stack trace
- **Recommendation:** Set `APP_DEBUG=false` in production immediately.

# ISSUE 3 — Admin Login Error

## 1. Issue Title
**Laravel Telescope Crashes Every HTTP Request (Including Admin Login) Due to Missing `telescope_entries` Table**

## 2. Error Category
**Infrastructure / Configuration / DB**

## 3. Technical Root Cause Analysis

Laravel Telescope is installed as a package and registered in `config/app.php` at line 180:

```php
App\Providers\TelescopeServiceProvider::class,
```

Telescope is configured in `config/telescope.php` with:

```php
'enabled' => env('TELESCOPE_ENABLED', true),  // defaults to true
'driver'  => env('TELESCOPE_DRIVER', 'database'),
```

**The `telescope_entries` table does not exist in the database.** The migration file exists at `database/migrations/2018_08_08_100000_create_telescope_entries_table.php` but was never run.

**Cascade of failure:**

1. Every HTTP request triggers Telescope watchers (query, request, model, cache, exception watchers — all enabled by default)
2. Telescope attempts to batch-insert watched events into `telescope_entries`
3. MySQL throws: `SQLSTATE[42S02]: Base table or view not found: 1146 Table 'asttrolok_live_db.telescope_entries' doesn't exist`
4. This exception occurs DURING the request lifecycle, corrupting the response
5. For the admin login specifically: The login form renders, but when the POST happens, the authentication succeeds internally, but the Telescope write crashes the response before the redirect can complete
6. The user sees either a 500 error page or a blank page — login appears to "fail"

**Evidence from logs:**

```
SQLSTATE[42S02]: Base table or view not found: 1146 Table 'asttrolok_live_db.telescope_entries' doesn't exist
(Connection: mysql, SQL: insert into `telescope_entries` (`batch_id`, `content`, `created_at`, `family_hash`, `type`, `uuid`) values (...))
```

This error is logged repeatedly for EVERY request, not just login.

## 4. Why This Error Occurs

1. **Telescope was added as a dependency** (`composer require laravel/telescope`)
2. **Service provider was registered** in `config/app.php`
3. **Telescope migration was never executed** (`php artisan telescope:install` or `php artisan migrate` was not run for this migration)
4. **No `TELESCOPE_ENABLED` env variable** was set — defaults to `true`
5. **Environment is `local`** — Telescope's filter function logs ALL events in local mode:
   ```php
   Telescope::filter(function (IncomingEntry $entry) use ($isLocal) {
       return $isLocal || ...;  // true for local env
   });
   ```

## 5. Which Layer Is Failing

| Layer | Status |
|-------|--------|
| **Frontend (Login form)** | ✅ Renders correctly |
| **Auth middleware** | ✅ Authentication succeeds |
| **Controller (LoginController)** | ✅ Processes credentials correctly |
| **Telescope ServiceProvider** | ❌ **FAILING** — tries to write to non-existent table |
| **MySQL** | ✅ Working (table genuinely missing) |

**Failing layer:** Telescope Service Provider (infrastructure package) corrupting request lifecycle

**Critical note:** This is NOT an auth bug — it's an infrastructure bug that manifests as a login failure because it kills every HTTP response.

## 6. SQL Fix

**Option A (Recommended — Disable Telescope):**
No SQL needed. Add to `.env`:
```
TELESCOPE_ENABLED=false
```

**Option B (If Telescope is wanted):**
```bash
php artisan migrate --path=database/migrations/2018_08_08_100000_create_telescope_entries_table.php
```

## 7. Backend Fix

**Applied fix:** Added `TELESCOPE_ENABLED=false` to `.env` file, then cleared config cache:

```bash
php artisan config:clear
php artisan cache:clear
```

**Alternative (more robust):** Conditionally register the service provider in `config/app.php`:

```php
// Replace unconditional registration with conditional
// In AppServiceProvider::register():
if (config('telescope.enabled')) {
    $this->app->register(\App\Providers\TelescopeServiceProvider::class);
}
```

## 8. Security Risk

- **🔴 HIGH.** This is a system-wide denial of service:
  - **ALL HTTP requests fail** — not just admin login
  - User login, API calls, page loads — everything affected
  - Stack traces exposed to all users via `APP_DEBUG=true`
  - Telescope was configured to log ALL events in local mode, including sensitive data (request parameters, queries, models)

2. ✅ Add `TELESCOPE_ENABLED=false` to `.env`

# ISSUE 4 — Access Denied on Support Tickets

## 1. Issue Title
**Missing `admin_support_manage` Permission in `sections` Table Causes 403 Access Denied for Admin Support Ticket Page**

## 2. Error Category
**Permission / Authorization / DB**

## 3. Technical Root Cause Analysis

The `AdminSupportController::index()` at line 40 performs:

```php
public function index(Request $request)
{
    $this->authorize('admin_support_manage');
    // ...
}
```

Laravel's `$this->authorize()` calls the Gate system. In this application, the authorization system works via the `sections` + `permissions` tables:

1. `sections` table stores permission names (e.g., `admin_supports_list`, `admin_support_manage`)
2. `permissions` table maps `role_id` → `section_id` → `allow` (1/0)
3. The Gate checks if the current user's role has `allow=1` for the requested section

**The `admin_support_manage` section did NOT exist** in the `sections` table. The existing support permissions are:

| id | name | caption |
|----|------|---------|
| 950 | admin_supports | Admin Supports |
| 951 | admin_supports_list | — |
| 952 | admin_support_send | — |
| 953 | admin_supports_reply | — |
| 954 | admin_supports_delete | — |
| 955-959 | admin_support_department_* | Department CRUD |
| 959 | admin_support_course_conversations | — |

The `admin_support_manage` permission was introduced in a migration (`2026_02_10_140009_add_admin_support_manage_permission.php`) that **failed to run** because it referenced a `key` column in the `sections` table that doesn't exist:

```
SQLSTATE[42S22]: Column not found: 1054 Unknown column 'key' in 'where clause'
(SQL: select exists(select * from `sections` where `key` = admin_support_manage))
```

The migration had a bug — it used `where('key', ...)` instead of `where('name', ...)`.

## 4. Why This Error Occurs

**Two cascading failures:**

1. **Migration bug:** The migration that should create this permission references the wrong column name (`key` instead of `name`)
2. **Migration never ran:** Because it failed, the permission was never inserted
3. **No fallback:** The controller uses `$this->authorize('admin_support_manage')` without a fallback — if the permission doesn't exist, the Gate returns `false`, resulting in a 403

Even if the migration had run correctly, the permission still needed to be **granted** to the admin role via an INSERT into the `permissions` table.

## 5. Which Layer Is Failing

| Layer | Status |
|-------|--------|
| **Route** | ✅ Working — route resolves to `AdminSupportController@index` |
| **Middleware (auth)** | ✅ Working — admin is authenticated |
| **Controller authorization** | ❌ **FAILING** — `$this->authorize('admin_support_manage')` returns 403 |
| **Gate/Policy** | ❌ **FAILING** — permission not found in sections table |
| **Database** | ❌ **FAILING** — section record missing + permission grant missing |

**Failing layer:** Authorization Gate backed by `sections` + `permissions` tables

## 6. SQL Fix

```sql
-- Step 1: Find the section_group_id for support permissions
SELECT section_group_id FROM sections WHERE name = 'admin_supports' LIMIT 1;
-- Result: (whatever group_id)

-- Step 2: Insert the missing permission
INSERT INTO sections (name, section_group_id, caption)
VALUES ('admin_support_manage', <group_id>, 'Admin Support Manage');

-- Step 3: Grant to Admin role (role_id=2)
INSERT INTO permissions (role_id, section_id, allow)
VALUES (2, (SELECT id FROM sections WHERE name = 'admin_support_manage'), 1);

-- Step 4: Grant to Support Role (role_id=9)
INSERT INTO permissions (role_id, section_id, allow)
VALUES (9, (SELECT id FROM sections WHERE name = 'admin_support_manage'), 1);
```

**Applied:** ✅ All 3 SQL statements executed. Permission created (id=3053) and granted to both roles.

## 7. Backend Fix Example

The broken migration at `2026_02_10_140009_add_admin_support_manage_permission.php` needs to be fixed:

```php
// BROKEN (references non-existent 'key' column):
if (!DB::table('sections')->where('key', 'admin_support_manage')->exists()) {

// FIXED (use correct 'name' column):
if (!DB::table('sections')->where('name', 'admin_support_manage')->exists()) {
```

## 8. Security Risk

- **Low direct risk** — the error is a false negative (denying legitimate access), not a false positive
- **Operational risk is HIGH** — admins cannot manage support tickets, causing support workflow to be completely blocked
- No data exposure since 403 is a clean denial

# ADDITIONAL FINDINGS — Secondary Issues Discovered During Investigation

## Secondary Issue A: `flow_type` Column NOT NULL Without Default

**Column:** `new_support_for_asttrolok.flow_type`
**Problem:** Defined as `VARCHAR(255) NOT NULL` with no default value. The controller sets `flow_type` in some code paths but not all. Any insert without explicit `flow_type` fails with `General error: 1364 Field 'flow_type' doesn't have a default value`.
**Fix Applied:** `ALTER TABLE new_support_for_asttrolok MODIFY COLUMN flow_type VARCHAR(255) NULL DEFAULT NULL`

## Secondary Issue B: `purchase_status` Column NOT NULL Without Default

**Column:** `new_support_for_asttrolok.purchase_status`
**Problem:** Same as above — NOT NULL without default. Not all support scenarios involve a purchase.
**Fix Applied:** `ALTER TABLE new_support_for_asttrolok MODIFY COLUMN purchase_status VARCHAR(255) NULL DEFAULT NULL`

## Secondary Issue C: Broken Migration File

**File:** `database/migrations/2026_02_10_140009_add_admin_support_manage_permission.php`
**Problem:** Uses `where('key', 'admin_support_manage')` but `sections` table has `name` column, not `key`. Migration will always fail.
**Status:** Fixed manually via direct SQL. Migration file should be corrected for future deployments.
